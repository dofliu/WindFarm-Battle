# 🎨 VISUAL WORKFLOW — 卡牌視覺工作流

> **這是本專案最重要的視覺設計文件。** 內含血淚得來的工作流決策。

---

## 🎯 核心原則

> **「AI 只畫插畫，模板做設計。」**

不要叫 AI 「畫一張完整的卡」——這會導致每張卡的 HUD、邊框、字體都不同。**業界標準做法**是：

1. **AI 生純插畫**（只有畫面，無任何 UI、文字、邊框）
2. **統一卡框模板**（一個設計師做好就重複用）
3. **程式自動套版**（從 `cards.json` 讀卡名、數據、能力，填入模板）

這就是萬智牌、爐石、寶可夢的做法。

---

## 📖 為什麼這樣決定？

### 失敗案例（我們踩過的坑）

開發者 Dof 用 GPT 生了 3 張卡（M06、F06、T05），看起來都很漂亮，但：

❌ **問題 1：不像卡牌**
- 每張看起來像「海報」、「資訊圖」、「人物照」
- 沒有清楚的卡牌結構（卡名區、能力區、數據條）

❌ **問題 2：沒有統一風格**
- M06 的 HUD 浮層是青色發光
- F06 的標註框是白底黑線
- T05 完全沒有任何 UI
- 三張根本不像同一套遊戲

❌ **問題 3：資訊密度錯誤**
- F06 那張資訊豐富到像「論文海報」
- 縮小到 88mm 卡片尺寸後完全看不清

### 根本原因

```
舊做法：「畫一張完整的卡」
       ↓
   AI 用想像力編造 HUD/邊框/標註
       ↓
   每張卡都長不同樣
```

```
新做法：「畫卡上的插畫」+ 統一模板
       ↓
   AI 只負責中央插畫，乾淨單純
       ↓
   模板提供統一框架
       ↓
   所有卡看起來是同一套
```

## 🎴 卡牌結構規範

每張卡都遵循以下版面（**模板自動處理**）：

```
┌─────────────────────────────────┐
│  [ID]  [TYPE]            [COST]  │ ← 頂部標籤條
├─────────────────────────────────┤
│  卡名 (大字)                      │
│  Subtitle / 英文標籤              │ ← 卡名區
├─────────────────────────────────┤
│                                 │
│                                 │
│       AI 純插畫位置              │ ← 主視覺
│       (4:3 比例)                 │
│                                 │
│                                 │
├─────────────────────────────────┤
│  [MW]    [%]    [m]              │ ← 數據條（從 cards.json）
├─────────────────────────────────┤
│  🛡️ 【能力1】描述...               │
│  ⚡ 【能力2】描述...                │ ← 能力區（從 cards.json）
│  📊 【能力3】描述...                │
├─────────────────────────────────┤
│  "flavor text..."                │ ← 風味文字
├─────────────────────────────────┤
│  ★★★★★          LEGENDARY        │ ← 底部
└─────────────────────────────────┘
```

**金色框 / 紫色框 / 銀色框** 由稀有度自動決定。

## 🎨 六大卡類視覺風格

| 卡類 | 視覺風格 | 範例氛圍 |
|---|---|---|
| ⚙️ **機組** | 真實 CG 寫實風 | 像產品型錄、廠商宣傳照 |
| 👷 **技師** | 半寫實人物插畫 | 像專業攝影，但有畫感 |
| ⚠️ **故障** | 工程示意圖 | 像 NREL 技術報告、IEC 文件 |
| 🔧 **功能** | 科技 HUD 線條 | 像 Iron Man HUD、SCADA 介面 |
| 🌤️ **天氣** | 氣象局發布圖 | 像 CWB 等壓線圖、衛星雲圖 |
| 📋 **合約** | 文件 / 標案風 | 像投標書封面、政府公文 |

## 📝 AI Prompt 三鐵則

無論用哪個 AI（Midjourney / DALL-E / Gemini / SD），prompt 都遵守：

### 鐵則 1：明確排除 UI 元素

```
--no text, no labels, no HUD overlays, no data panels, 
no UI elements, no badges, no logos, no watermarks, 
no borders, no frames, no callout labels, no annotations
```

### 鐵則 2：引用真實技術文件風格

不要說「畫一個齒輪箱故障」，要說：

```
Technical engineering illustration in the style of 
NREL technical reports / IEC 61400 documentation / 
Vestas product brochures
```

### 鐵則 3：指定構圖比例與留白

```
--ar 4:3            ← 卡牌插畫專用比例
clean composition with space at top and bottom 
                    ← 留出可被模板邊緣覆蓋的空間
```

完整 prompt 模板見 `CARD_PROMPTS.md`。

## 🛠️ 套版工具規格（Sprint 4 要做的）

### 輸入

```typescript
type CardArtInput = {
  cardId: string;          // 例：'M07'
  artPath: string;         // 例：'./art/M07-tianjing.png'
};
```

### 處理流程

```
1. 讀取 cards.json[cardId] 取得完整資料
2. 載入插畫圖檔 artPath
3. 套用模板（含金色框、卡名、能力、數據條）
4. 渲染為 React 元件 或 導出為 PNG/PDF
```

### 輸出

```typescript
type CardOutput = {
  preview: ReactElement;   // 螢幕預覽
  png: Blob;               // 300DPI 印製用
  pdf: Blob;               // 批次印製版
};
```

### 批次處理

```bash
npm run cards:build       # 讀取 ./art/ 中所有 *.png，產出 ./output/cards/*.png
npm run cards:pdf         # 產出可印製的 PDF（每頁 9 張卡）
```

## 🎯 為什麼這個工作流很重要

### 對學生（教學）
- 統一視覺降低認知負荷
- 故障卡的工程示意圖風格 = 學習工程語彙
- 像真實 IEC 文件 = 學生熟悉產業視覺

### 對教師（你）
- AI 生圖快速，不必每張找美工
- 修改規則只需更新 cards.json，視覺自動跟著變
- 印製版本容易產出

### 對研究（論文）
- 視覺一致 = 看起來像「真正的遊戲」
- 投稿截圖會比 4 不像的圖好看 10 倍
- 可作為「AI 輔助教學媒材製作」研究素材

### 對未來
- 多語版本只需重新填卡名與能力描述
- 風格更新只需改模板一次
- 新增 Tier 2 卡片無痛擴充

## ✅ 視覺工作流驗收標準

當你（Claude Code）做完 Sprint 4 卡牌模板系統時，**必須通過以下測試**：

- [ ] 給定任一卡 ID + 純插畫，能在 1 秒內產出完整卡牌
- [ ] 47 張卡渲染後，視覺一致度肉眼看 95%+
- [ ] 卡牌庫頁面可一次顯示全部 47 張卡（無視覺斷層）
- [ ] 印製版 300DPI PNG 解析度足夠
- [ ] 可導出含背面的 PDF（兩面對齊）
- [ ] 修改稀有度（cards.json）後，邊框顏色自動更新
