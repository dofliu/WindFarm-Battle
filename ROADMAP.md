## 🔄 路線修訂（2026-05-21，Dof 確認）

重構接手後與 Dof 重新對齊優先順序，本節**優先於下方原始時程**：

1. **機制優先，視覺/印刷往後**：先把「能玩 ＋ 規則正確 ＋ 平衡站得住 ＋ 能記錄數據」做穩。原本的「視覺資源產出 / 卡牌美術模板系統」**降級延後**，等機制與平衡定案後才做；紙本印刷更後面。
2. **新增「天氣對抗模式（同題競賽）」**：非對戰模式。電腦生成同一串環境條件（風速／天氣／設備／故障），多名玩家（或單人 vs AI 基準）各自在限定回合內最佳化收入，最後比分。架構上 `core/` 設計為「模式無關」，對戰與同題競賽共用同一規則引擎；同一亂數種子＝同一道題目。
3. **動畫／音效列為後期增強**：PTCG 式出牌／出招動畫與音效，待機制定案後再做。UI 從一開始就以 core 吐出的「事件流」驅動，動畫掛在事件上即可。
4. **研究用統計／遙測**：分兩塊——模擬分析（僅供平衡，**不可當論文實驗數據**）、真人對局遙測（結構化記錄，可匯出供論文）。同題競賽模式變數受控，最適合產生可比較的研究數據。收學生資料涉受試者同意／IRB。

**修訂後開發順序**：S1 骨架 → S2 core(27張＋對齊) → S3 B區20張 → S4 模式系統(對戰＋同題競賽) → S5 UI(可玩) → S6 模擬器＋遙測 →〔機制/平衡定案〕→ S7 卡牌美術模板 → S8 動畫音效 → S9 手機 App。

## 📅 時程規劃

### Sprint 0：交接準備（1 週）

**目標**：把所有設計資產整理為 Claude Code 可接手的形式

- [x] 設計文件整理（DESIGN.md、CLAUDE.md）
- [x] 卡牌資料結構化（cards.json）
- [x] 原型保留為設計史
- [ ] **建立 GitHub Repo**
- [ ] 第一次與 Claude Code 對話：分析重構計畫

### Sprint 1：技術骨架（1 週）

**目標**：建立可運行的開發環境

```bash
- npm init + Vite + React + TS
- TailwindCSS 設定
- Zustand store 起手式
- Vitest 測試環境
- ESLint + Prettier 規則
- 基本的 Git workflow（含 hooks）
```

**產出**：能 `npm run dev` 並看到「Hello WindFarm Battle」

### Sprint 2：Core 層重構（2 週）

**目標**：把原型的遊戲邏輯重構為純 TypeScript

```
src/core/
├── types.ts          # GameState、Card、Player 等型別
├── cards.ts          # 從 cards.json 載入並提供查詢介面
├── game-state.ts     # 狀態與動作（不可變更新）
├── rules-engine.ts   # 規則邏輯（風速、發電、故障處理）
└── ai/
    ├── evaluator.ts  # 評估函式
    ├── strategy.ts   # 局勢判斷
    └── difficulty.ts # 難度系統
```

**驗收標準**：
- 80%+ 單元測試覆蓋
- 對齊測試通過（重構後行為與原型一致）
- 可獨立執行 1000 場模擬（CLI 模式）

### Sprint 3：UI 遷移（2 週）

**目標**：把 HTML 原型的視覺重做為 React 元件

```
src/ui/
├── components/
│   ├── Card.tsx           # 卡牌元件（多種尺寸）
│   ├── PlayerArea.tsx     # 玩家區
│   ├── HandArea.tsx       # 手牌區
│   ├── WindDisplay.tsx    # 風速面板
│   ├── LogPanel.tsx       # 戰況記錄
│   └── AIThoughtPanel.tsx # AI 思考過程
├── containers/
│   ├── GameContainer.tsx  # 遊戲主畫面
│   ├── LibraryView.tsx    # 卡牌庫
│   └── SimulationView.tsx # 模擬模式
└── styles/
```

**驗收標準**：
- 視覺與原型一致度 90%+
- 響應式設計（手機、平板、桌機）
- 所有互動可用

### Sprint 4：視覺資源產出（⏸ 已依路線修訂延後，見頂端「路線修訂」）

> 此階段移到「機制與平衡定案後」才執行（修訂後順序中的 S7）。以下內容保留供日後參考。

**目標**：用 AI 為 47 張卡產生統一風格的插畫

- 設計 Midjourney/DALL·E prompt 模板
- 為每張卡生成 4-5 個候選圖
- 設計師 / Dof 篩選與微調
- 整合到卡牌元件

**參考標準**：M07 天鯨的金色傳奇邊框風格

### Sprint 5：行動 App（2 週）

**目標**：Web 版包裝為 iOS + Android App

```bash
- npm install @capacitor/core @capacitor/cli
- npx cap init
- npx cap add ios
- npx cap add android
- Build & deploy
```

**處理事項**：
- 觸控優化（手指比滑鼠粗）
- 螢幕方向適配
- App icon、splash screen
- iOS / Android 商店審核準備

### Sprint 6：學生實測（4 週）

**目標**：在 DOF Lab 與 NCUT 風電課程中試行

- 召募 10-20 名學生
- 每人玩 5 場以上
- 記錄回饋（問卷 + 訪談）
- 分析資料、調整平衡

**研究價值**：
- 教學遊戲化效果量化研究
- 可成為論文核心資料

### Sprint 7：多人連線（4-6 週，選用）

**目標**：支援線上對戰

技術選擇：
- 後端：Node.js + WebSocket（Socket.IO）
- 雲端：Firebase Realtime Database
- 認證：Firebase Auth / GitHub OAuth

功能：
- 隨機配對
- 私人房間（給朋友、課堂）
- 戰績統計
- 簡單排行榜

## 💼 商業考量

### 模式 A：免費教育版（主軸）

- GitHub 開源
- 學校 / 教師免費下載
- DOF Lab 持續維護
- 收益：產學合作專案

### 模式 B：學校採購版（次要）

- 進階版含教師後台
- 收費 NT$2,000-5,000 / 班 / 學期
- 包含教學手冊與技術支援

### 模式 C：國際發行（選用）

- 翻譯為英文、日文、德文
- 上架 Steam / App Store
- 收費 USD 5-10 一次性

**選擇方向：** 主要 A，搭配 B 維持永續經營。C 視情況。

## ⚠️ 風險與緩解

| 風險 | 影響 | 緩解策略 |
|---|---|---|
| 學生覺得單調 | 高 | 持續加新卡、加入競爭模式 |
| 平衡崩壞 | 中 | 1000 場模擬驗證機制 |
| 視覺資源不足 | 中 | AI 生成、學生協作 |
| Claude Code 重構失敗 | 中 | 保留原型作為 fallback |
| 投稿論文被拒 | 低 | 多投幾家、改進實驗設計 |
| 沒有時間維護 | 高 | 招碩士生協助、開源吸引貢獻者 |

## 🤝 給未來貢獻者

如果你接手這個專案：

1. **不要打破現有設計**：改之前先讀 DESIGN.md
2. **保持向後相容**：舊版存檔要能用
3. **記錄決策**：每次重大改動寫進 DECISIONS.md
4. **尊重原型**：v1-v4 的 HTML 是歷史，不要刪
5. **持續測試**：1000 場模擬不是擺好看的
