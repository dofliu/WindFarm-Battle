## Part 1：設計哲學

### 核心信念
> **"降低嘗試門檻，提高深度可及性"**

這個遊戲服務三類人：
- **完全外行**（朋友、家人、新生）：能 5 分鐘上手，玩得開心
- **修課學生**：玩遊戲就學到 IEC 61400、SCADA、CMS 等概念
- **業界專業人士**：能在策略中看到實際運維工作的縮影

設計上採用 **「漸進式揭露」（Progressive Disclosure）**：
- 遊戲卡正面保持乾淨（IEC 編號只是右下角小標記）
- 教學深度放在「教學參考卡」上（獨立 21 張）
- 不強迫學生面對標準術語

### 設計鐵則

1. **教學遊戲第一鐵則**：第一次接觸 ≤ 3 種卡，5 分鐘上手
2. **不破壞代入感**：規則服務體驗，而非反過來
3. **平衡優先**：所有策略都應有 30-37% 勝率（依模擬驗證）
4. **沒有絕對贏家**：任何卡牌組合都應有應對方法
5. **教學情境真實**：故障類型、修復流程、IEC 標準都來自實際業界

## Part 2：遊戲總覽

### 基本資訊

| 項目 | 規格 |
|---|---|
| 玩家數 | 1-2 人（單機）/ 未來支援多人 |
| 遊戲時間 | 15-30 分鐘 |
| 牌組大小 | 27 張（Tier 1）/ 47 張（v4）/ 80+ 張（未來） |
| 回合數 | 12 回合 |
| 雙方手牌上限 | 7 張 |
| 場上機組上限 | 3 台（可替換） |

### 勝利條件

**主要勝利**：第 12 回合結束時，累計 MWh 較高者勝

**未來擴充**：
- 停機宣判（對手所有機組可用率 = 0% 連續 2 回合）
- 合約判定（完成隱藏目標獲得獎勵）

### 回合流程（8 階段）

```
1. 風速階段：擲 1d6 決定本回合風速
2. 天氣階段：處理持續中的天氣效果
3. 抽牌階段：雙方各抽 1 張
4. 行動階段 P1：玩家 1 使用 2 動作
5. 行動階段 P2：玩家 2 使用 2 動作
6. 結算階段：計算雙方 MWh 並加總
7. 維護階段：故障倒數、自動修復、特殊效果
8. 換手階段：先後手交換
```

## Part 3：核心機制

### 3.1 風速系統（1d6 骰子）

| 骰點 | 風速 | 效率係數 | 標籤 |
|---|---|---|---|
| 1 | 0 m/s | ×0.0 | 無風停機 |
| 2-3 | 5 m/s | ×0.4 | 低風 |
| 4-5 | 10 m/s | ×1.0 | 額定 ✅ |
| 6 | 20 m/s | ×0.7 | 高風（風險） |
| 6+6 | 25 m/s | ×0.0 | 🌀 颱風！ |

擲到 6 必須再擲一次：再次 6 = 颱風事件，所有機組強制停機。

### 3.2 發電公式

```
單機 MWh = MW × 風速係數 × (可用率 / 100)
玩家本回合 MWh = Σ 所有機組 MWh × 各種加成
```

**加成可能來源：**
- M07 天鯨光環：所有機組 +1 MW
- B02 智慧電網：固定 +3 MWh
- FN06 緊急投標：本回合 ×1.5
- W04 電網爆發：本回合 ×2

### 3.3 故障系統

故障按嚴重度（Sev.1-5）分級：

| Sev | 例 | 可用率扣分 | 修復回合 | 動作費 |
|---|---|---|---|---|
| 1 | 感測器異常 | -5% | 1（自動） | 1 |
| 2 | 液壓漏油 | -10% | 2 | 1 |
| 3 | 葉片損傷 | -20% | 2 | 2 |
| 4 | 齒輪箱磨損 | -30% | 3 | 2 |
| 5 | 主軸承故障 | -50% | 3 | 2 |

**特殊機制：**
- **連鎖機率**（F06、F07）：20-30% 機率傳染到鄰機
- **擴散效應**（F03）：不修復則每回合 -5%
- **專屬克制**（F04、F06）：必須有對應技師才能修復

### 3.4 技師克制矩陣

| 故障 | 必要克制 | 可選克制 |
|---|---|---|
| F01 感測器異常 | — | T01, T04 |
| F02 偏航誤差 | — | T01 |
| F03 液壓漏油 | — | T01 |
| F04 葉片損傷 | **T02 葉片檢查員** | T08 無人機操作員 |
| F05 變槳系統失效 | — | T04 |
| F06 齒輪箱磨損 | **T03 齒輪箱技師** | — |
| F07 主軸承故障 | — | T03 + T06 |

**關鍵原則：** 故障被克制時 = 該回合結束時自動修復

### 3.5 動作費經濟

每回合 2 動作費（基礎），可累積至 4：
- T07 資深主管：+1 動作 / 回合
- FN04 預支動作：下回合 +1（可累積）
- T09 研發總監：每用一張功能卡 +1 動作

**費用調整：**
- T07 在場：技師卡 -1（最低 1）
- T09 在場：功能卡 -1（最低 0）

## Part 5：AI 系統設計

### 5.1 評估函式架構

**核心理念**：對每個可能行動打分，選最高分

```
總體流程：
1. generateActions(playerId) → 所有可能行動列表
2. 對每個 action 呼叫 evaluate*Play() 計算分數
3. 排序，依難度選擇：
   - Easy：50% 機率選 top-3 隨機
   - Medium：25% 機率選第二名
   - Hard：永遠選第一名
```

### 5.2 評估函式設計

#### A. 評估部署機組

```typescript
function evaluateTurbinePlay(card, player, strategy): number {
  // 期望發電：MW × 平均風係數 × 可用率 × 剩餘回合
  let expectedMWh = card.mw * 0.65 * (card.avail/100) * strategy.roundsLeft;
  let score = expectedMWh * 1.5;
  
  // 動作費懲罰
  score -= card.cost * 4;
  
  // 邊際遞減（已有機組越多越不需要）
  score *= Math.pow(0.75, player.turbines.length);
  
  // 階段調整
  if (strategy.phase === 'early') score *= 1.4;
  if (strategy.phase === 'late') score *= 0.4;
  
  // 弱點懲罰
  if (card.special === 'big-fail') score -= 8;
  
  // 落後時更需要產能
  if (strategy.position === 'losing') score *= 1.2;
  
  return score;
}
```

#### B. 評估施加故障

```typescript
function evaluateFaultPlay(card, target, player, strategy): number {
  let score = 0;
  
  // 關鍵：檢查對手 counter
  if (card.required && !opponentHasRequired) {
    score += 35; // 對手無法修復，超高分
  }
  if (opponentHasCounter) {
    score -= 25; // 會被立刻修，不值得
  }
  
  // 傷害期望：drop% × 目標MW × 風係數 × 持續回合
  const damage = target.mw * 0.65 * (card.drop/100) * card.rounds;
  score += damage * 2.5;
  
  // 連鎖機率期望值
  if (card.cascade) score += damage * card.cascade * 1.5;
  
  // 階段調整
  if (strategy.phase === 'late') score *= 1.3;
  
  return score;
}
```

### 5.3 局勢感知（Strategy Awareness）

每次決策前都會評估：

```typescript
type Strategy = {
  phase: 'early' | 'mid' | 'late';  // 開局/中盤/終局
  position: 'winning' | 'even' | 'losing';  // 領先/均勢/落後
  myMWhPerRound: number;
  oppMWhPerRound: number;
  scoreDiff: number;
  roundsLeft: number;
}
```

判斷規則：
- **開局**（回合 1-4）：建設導向，加倍部署機組價值
- **中盤**（回合 5-8）：均衡攻防
- **終局**（回合 9-12）：攻擊加成，不部署新機組
- **落後 > 15**：放手一搏，攻擊加成 1.4×
- **領先 > 15**：保守打法，攻擊降為 0.85×

### 5.4 難度設計

| 難度 | 行為 | 適合對象 |
|---|---|---|
| 😊 入門 | 50% 機率選 top-3 隨機 | 新生、新手 |
| 😐 中級 | 25% 機率選次優 | 中等程度玩家 |
| 😈 高手 | 永遠選最優 | 挑戰真實玩家 |

#### 天氣對抗模式（同題競賽）— 草案，待細化

非對戰模式：系統以**同一亂數種子**生成一連串環境條件（風速、天氣、設備狀況、故障事件），讓所有參與者面對**完全相同的一道題目**。各玩家（或單人對 AI 基準）在限定回合內，獨立做出最佳因應、追求最高累計收入，最後比結果。

設計動機：(1) 把焦點從「打對手」轉向「對抗環境、做好運維決策」，更貼近真實工作；(2) 變數受控、人人同題，最適合作為**研究數據**來源（見路線圖的遙測機制）；(3) 天然支援單人遊玩與課堂同步競賽。

**待定的設計問題**：同題模式中玩家之間是否仍可互相施加故障（半對抗），或純粹「各自 vs 環境」？此決定計分方式，於實作該模式時定案。

> 工程含意：`core/` 規則引擎須「模式無關」，對戰與同題競賽共用同一套規則，僅勝負/計分控制器不同；同題＝同 seed。詳見 REFACTOR_PLAN.md 的 D7。

## Part 8：技術架構建議

### 8.1 程式碼分層

```
┌─────────────────────────────────────┐
│  UI Layer (React Components)        │ ← 介面，可換不同平台
├─────────────────────────────────────┤
│  State Management (Zustand)         │ ← 狀態管理
├─────────────────────────────────────┤
│  Core Logic (Pure Functions, TS)    │ ← 共用邏輯，無 UI 依賴
│  ├── cards.ts                       │
│  ├── game-state.ts                  │
│  ├── rules-engine.ts                │
│  ├── ai-evaluator.ts                │
│  └── simulation.ts                  │
├─────────────────────────────────────┤
│  Data (JSON)                        │ ← 卡牌資料
│  └── cards.json                     │
└─────────────────────────────────────┘
```

**關鍵原則：** `core/` 是純函式 + TypeScript，**完全不依賴 UI 框架**。這樣同一份邏輯可用於：
- Web 版（React）
- 手機 App（React Native 或 Capacitor）
- CLI 模擬器（Node.js）
- 未來的桌機版（Tauri / Electron）

### 8.2 狀態管理

使用 **Zustand**（比 Redux 簡單，比 Context 更高效）：

```typescript
type GameState = {
  round: number;
  wind: Wind;
  currentPlayer: 0 | 1;
  actionsLeft: number;
  players: [Player, Player];
  log: LogEntry[];
  weatherEffects: WeatherEffect[];
  // ...
};

const useGameStore = create<GameState>((set) => ({
  // initial state
  // actions
  playCard: (handIdx: number, options?: PlayOptions) => set(...),
  endTurn: () => set(...),
  // ...
}));
```
